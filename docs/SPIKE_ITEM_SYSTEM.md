- `core/event_bus.gd` — inventory/item-use signals
- `data/items/` — data-driven item resources

## APIs & Data Shapes (summary)

- ItemDefinition: `id`, `name_key`, `description_key`, `icon`, `item_type`, `stackable`, `usable`, `modifiers: Array[ModifierDefinition]`, `use_action`
- ModifierDefinition: `target` (e.g., `resource.stamina`, `attribute.health_max`), `operation` (`add|mul|override`), `value`, `condition` (`on_use|equipped|always`)
- ItemInstance: `definition`, `quantity`, `custom_state`
- ItemRegistry: `get_item(id)`, `list_items()`
- InventorySystem: `add_item(entity_id,item_id,qty)`, `remove_item(...)`, `request_use_item(entity_id,item_id)`, save/load hooks
- ResourceSystem: `add_resource`, `set_resource`, `get_resource_amount`, `can_pay`, `apply_cost`
- CharacterSystem / AttributeResolver: registers entities, manages modifiers and derived attributes

## Integration Map (how flows currently work)

- Items are defined as `.tres` in `data/items/` and loaded by `ItemRegistry`.
- UI (Inventory/Shop) emits use/buy requests via `EventBus`.
- `Inventory.request_use_item` emits `EventBus.item_use_requested`.
- `player.gd` currently listens and applies `on_use` modifiers to `ResourceSystem` (resource-only, additive/override; `mul` not implemented).
- `InventorySystem` listens for `item_use_success` to decrement consumables.

## Missing / Incomplete Pieces

- No central `ModifierApplicator` that routes item modifiers to correct systems (resource vs attribute) and handles `mul` semantics.
- `player.gd` only processes resource targets; attribute/skill targets and `mul` not handled.
- No unified temporary modifier API for timed buffs from items.
- Combat integration is not present for item effects (out of scope for this spike).
- Tests are missing for `mul`, attribute modifiers, and save/load round-trip for items with `custom_state`.

## Risks & Assumptions

- Assumes autoloads: `Items`, `Inventory`, `Resources`, `Characters`, `EventBus` exist and are named accordingly.
- Assumes Godot 4.x API compatibility in project.
- Modifier `target` strings must follow conventions (`resource.<id>`, `attribute.<id>`); mismatches cause failures.
- Combat effects are not validated here; only resource and attribute interactions.

## Plan (concrete steps)

1. Create `core/items/modifier_applicator.gd` (autoload)
   - Purpose: centralize applying `ModifierDefinition` effects to `ResourceSystem` and `CharacterSystem`.
   - Subscribe: `EventBus.item_use_requested`.
   - API: `apply_item_use(entity_id:String, item_def:ItemDefinition) -> bool`.

2. Update `core/characters/player.gd`
   - Delegate item use handling to `ModifierApplicator` and emit `EventBus.item_use_success`/`item_use_failed` based on result.

3. Implement `mul` semantics and routing
   - Files involved: `core/items/modifier_definition.gd`, `core/items/modifier_applicator.gd`, `core/resources/resource_system.gd`, `core/characters/character_system.gd`.
   - Semantics: `add` (add to current), `mul` (scale current by factor), `override` (set absolute).
   - Routing: `resource.<id>` → `ResourceSystem`; `attribute.<id>` → `CharacterSystem` via temporary modifier API.

4. Add tests & spike scene
   - Add `test/test_item_use_flow.gd` with unit tests for add/use/consume, `mul` behavior, attribute modifier application, and save/load round-trip.
   - Add `test/scenes/spike_item_use.tscn` manual runner exercising UI → Inventory → Use flow.

5. Persistence check
   - Ensure `SaveSystem` calls `Inventory.get_save_state` and `Inventory.load_save_state` during save/load.
   - Add a save/load test for items with `custom_state`.
